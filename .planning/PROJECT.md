# WonderWaltz

## What This Is

WonderWaltz is a multi-platform Disney trip planner (iOS, Android, backend, marketing website) that generates personalized, day-by-day itineraries for Walt Disney World guests. A user enters their trip details — dates, guests with ages and mobility/sensory/dietary constraints, budget tier, on-property status, transportation, and must-do rides — and WonderWaltz produces a time-blocked schedule for each park day: which rides to hit in what order, when to buy and book Lightning Lanes, where and when to eat, and how to get the most from every day including budget hacks. v1 launches on WDW only; Disneyland and Disneyland Paris are post-v1.

## Core Value

**A first-time Walt Disney World visitor gets a plan that feels like a Disney expert made it for them.** Every tradeoff in design, engineering, and content should reinforce this promise. If the generated plan does not feel hand-crafted by someone who knows the parks deeply, nothing else matters.

### Active

#### Core planning
- [ ] User can create a trip with dates, per-guest profiles (age, mobility, thrill/sensory tolerance, dietary restrictions), budget tier (Pixie Dust / Fairy Tale / Royal Treatment), on-property status + specific resort, transportation mode, must-do rides, meal preferences, and park hopper choice
- [ ] User can view a full, time-blocked, per-day schedule for their trip, including rides, meals, shows/parades/fireworks, rest blocks, and Lightning Lane booking reminders
- [ ] Schedule is generated by a hybrid engine: a deterministic solver builds the time blocks using live + forecast data; Claude adds the narrative, per-item tips, and budget hacks
- [ ] Schedule adapts on demand via a "Rethink my day" button that re-runs the solver with current time and remaining items
- [ ] Full trip (schedule, catalog subset, walking graph, static maps) works offline on device after purchase

#### Data
- [ ] Park hours, attraction metadata, dining, shows, and wait times ingest automatically from queue-times.com (primary) and themeparks.wiki (secondary)
- [ ] Historical wait times accumulate in a Timescale hypertable from day 1
- [ ] Wait-time forecasts are generated from a statistical baseline (ride × day-of-week × hour × crowd-level bucket) with an explicit confidence label
- [ ] Weather forecast (OpenWeather) is fetched per trip day and feeds the solver (packing list, rain contingencies)

#### Mobile apps
- [ ] iOS app (native SwiftUI, iOS 17+) with onboarding, trip wizard, plan view, packing list, settings, paywall
- [ ] Android app (native Kotlin + Jetpack Compose) with feature parity to iOS via OpenAPI-generated clients
- [ ] Both apps support offline-first use in parks with flaky cell coverage
- [ ] Both apps support push notifications for Lightning Lane booking windows and "walk to X now" schedule events
- [ ] Both apps support Live Activities (iOS) / ongoing notification (Android) showing the next item in today's plan
- [ ] Both apps include a countdown-to-trip widget ("X days until your Disney trip")

#### Monetization
- [ ] Free tier shows a full Day 1 schedule and blurred Days 2–N as a teaser
- [ ] Per-trip consumable unlock via StoreKit 2 (iOS) and Google Play Billing (Android), receipt validation and entitlement via RevenueCat
- [ ] Single SKU: $9.99 per trip unlock at launch
- [ ] Affiliate packing list (Amazon Associates) curated by trip context: rain forecast, temperature, youngest-guest age, month of year

#### Accounts + auth

- [ ] Upgrade flow to Sign in with Apple or Google required before purchase; upgrade merges the anon account preserving any draft trip
- [ ] User can delete their account and all associated trip data (GDPR / CCPA compliance)

#### Website
- [ ] Marketing site (Next.js App Router on Vercel) with landing, how-it-works, pricing, 3–5 SEO park guides, FAQ, privacy, terms, and an "unofficial fan app" disclaimer on every page
- [ ] Admin panel (gated route in the same Next.js app) with catalog CRUD, crowd index overrides, user lookup, entitlement grants, plan regeneration, and LLM cost dashboards

### Out of Scope

#### v1 — explicit non-goals (anti-features)

- **In-app booking of Lightning Lanes, dining, or park tickets** — We recommend and deep-link to Disney's official app. We never hold payment details for Disney services. Legal simplicity and trust.
- **Storing Disney account credentials or integrating with My Disney Experience accounts** — We never ask for a user's Disney password. We never scrape their MDE account. Pure external recommendations only. Privacy and trust non-negotiable.
- **Real money movement in any savings or budgeting feature** — The future Saver Planner (v1.1) tracks goals only. No bank transfers, no held funds, no gift card custody beyond affiliate links. Becoming a money transmitter is not a viable path for a solo founder.
- **Ad-supported free tier** — Monetization is purchase + affiliate only. No banner ads, no interstitials, no sponsored itineraries.
- **Crowdsourced user-generated plans marketplace** — This is not a social platform. No shared plan feeds, follower graphs, or comment systems. Focus stays on personal trip planning.

#### v1 — deferred to later versions

- **Disneyland Anaheim and Disneyland Paris** — WDW only at launch. Added resorts come in v1.1+ once the engine is validated on one resort.
- **Dining reservation watchlist with push alerts** — High user value but requires a new data source (Disney's public dining endpoint), a dedicated polling worker, notification infrastructure, and IP lawyer review. v1.1.
- **Park crowd calendar (public web page)** — Natural SEO magnet. We already compute a crowd index for the solver, so exposing it is cheap. v1.1.
- **Home Screen / Lock Screen widget showing next item in today's plan** (iOS WidgetKit + Android Glance) — Delightful, learns its own platform quirks. v1.1.
- **Apple Watch / Wear OS companion + complication** — Part of a v1.1 watch companion milestone.
- **Saver Planner** — In-app goal tracker with budget pocket progress (goal-only, no real money). Pairs with the Pixie Dust / Fairy Tale / Royal Treatment tiers. v1.1.
- **Shareable read-only web trip view** — So family members can see the plan without installing the app. v1.1.
- **Spanish + Portuguese localization** — LATAM is a major WDW visitor base. Architecture supports i18n from day 1; translations ship in v1.1.
- **ML-based wait-time forecast** — The v1 baseline (bucketed median) is explicitly a placeholder. Revisit after 8+ weeks of historical data.
- **Own park-data scraper** — Kept as a kill-switch behind a legal + ToS review. Only activated if queue-times.com + themeparks.wiki become unreliable.
- **Character meet-and-greet tracking, autograph books, PhotoPass / Memory Maker integration** — Fan-community features. Scope creep risk. Deferred indefinitely.

## Context

**Market opportunity.** Existing Disney trip-planning tools leave significant room for something better. Disney's own Genie+ / Lightning Lane Multi Pass is transactional and clinical, not warm or personalized. TouringPlans is statistically rigorous but has a dated interface and a high learning curve. Mouse Dining focuses narrowly on reservations. Several mobile apps layer over Disney's app without solving the actual planning problem. None combine AI-personalized itineraries grounded in real wait-time data with budget optimization and a stress-free, hand-crafted feel. WonderWaltz targets the gap: a product that feels like it was made by a Disney expert specifically for you, cheaply enough that a first-timer will pay $10 to trust their vacation to it.

**Why a first-time visitor is the primary persona.** First-timers have the highest planning stress, the most to lose from a bad day, and the strongest motivation to pay for help. Repeat visitors already have mental models and shortcuts; first-timers do not. The core value statement ("feels like a Disney expert made it for them") is tuned specifically to this persona. Repeat visitors are a welcome bonus, not the design target.

**Data is the biggest technical risk.** Disney publishes no official API. Primary data comes from queue-times.com (live waits + hours) and themeparks.wiki (fallback + scheduled entertainment). Both are licensed for use with attribution and have been stable for years, but any reliance on them is a contract we do not control. Mitigations: local caching with longer TTLs than polling cadence, cross-source validation, Sentry alerts on ingestion lag, and a kept-but-unbuilt scraper fallback under legal review.

**Solo build reality.** The product is being built by one person plus AI coding agents. Every architectural choice favors managed services (Supabase, Railway, Vercel, Upstash, RevenueCat), one primary language across the backend + website (TypeScript), mirrored mobile architectures (SwiftUI ↔ Jetpack Compose) with OpenAPI-generated clients so parity is enforced at build time, and strict phasing so platforms are built in sequence, not parallel. There is no parallel track until Phase 5 of the build sequence.

**Design is a differentiator, not an afterthought.** The existing Disney-adjacent app category is visually generic. WonderWaltz will deliberately avoid "AI slop" aesthetics (Inter font, purple gradients, three-column feature grids, cookie-cutter SaaS polish). The repo already contains two strong design agents: `ui-designer` (creator) produces the system and specs, `ui-ux-designer` (research-backed critic with NN Group citations and explicit anti-generic-SaaS anti-patterns) reviews every UI surface. The brand direction will be selected from three explorations (vintage travel poster, warm modern minimalism, painterly whimsy) by the design agent with a recommendation.

**Legal posture is unofficial fan app.** "WonderWaltz" is Disney-adjacent phonetically without using the trademark. Every surface carries a clear disclaimer. Park and attraction names are used descriptively under nominative fair use. Zero Disney trademarked imagery. An IP lawyer review is a hard gate before public launch.

## Constraints

- **Platforms**: iOS (native SwiftUI, iOS 17+), Android (native Kotlin + Jetpack Compose), Backend (TypeScript + NestJS on Node 20), Website (Next.js App Router on Vercel). Launching simultaneously.
- **Backend stack**: NestJS + Drizzle ORM + Postgres (Supabase) + Redis (Upstash) + TimescaleDB extension. Workers via BullMQ.
- **Hosting**: Railway (api + workers), Supabase (Postgres + Auth + Storage), Upstash (Redis), Vercel (website). Managed everything.
- **LLM**: Anthropic Claude via the claude-api skill, with prompt caching, Haiku fallback for cheap narrative operations, and per-trip cost telemetry. Target p95 ≤ $0.20 per generated plan.
- **Monetization**: StoreKit 2 + Google Play Billing consumable per-trip IAP only. RevenueCat as the single source of truth for entitlements. Single $9.99 SKU at launch. No ads.
- **Auth**: Supabase Auth, anonymous-first, upgrade to Apple / Google before purchase.
- **Team**: Solo founder + AI coding agents. No parallel platform tracks. Scope ruthlessly capped.
- **Legal**: Unofficial fan app, clear disclaimer on every surface, zero Disney imagery, IP lawyer review before public launch.
- **i18n**: English at launch; all strings externalized from day 1 so Spanish + Portuguese can ship in v1.1 as drop-in translations.
- **Accessibility**: WCAG 2.2 AA baseline, enforced by the `ui-ux-designer` agent on every UI PR.
- **Data sources**: queue-times.com (primary) and themeparks.wiki (secondary) under their public terms. No own scraper in v1. No scraping of Disney proper in v1.
- **Privacy**: GDPR + CCPA. Anonymous account default; explicit consent on upgrade. Guest profiles (which include children) are trip attributes on the owner's account, not child accounts. Explicit deletion endpoint.
