    - `loader.getGraph(): WalkingGraph` returns the cached graph; if called before init, throws.
    - Plan-generation orchestrator (plan 03-16) uses `loader.getGraph()` per solve — NEVER queries the DB during solve.
    - Test with a DB spy: after onModuleInit, 10 subsequent `getGraph()` calls produce ZERO additional DB invocations.
  </behavior>
  <action>
    1. Create `plan-generation.module.ts` (minimal — will be extended in 03-16):
       ```ts
       @Module({ providers: [WalkingGraphLoader], exports: [WalkingGraphLoader] })
       export class PlanGenerationModule {}
       ```
    2. `walking-graph.loader.ts`:
       ```ts
       @Injectable()
       export class WalkingGraphLoader implements OnModuleInit {
         private graph?: WalkingGraph;
         constructor(@Inject(DB_TOKEN) private db: DbExecutable) {}
         async onModuleInit() {
           const rows = (await this.db.execute(sql`
             SELECT from_node_id::text, to_node_id::text, seconds FROM walking_graph
           `)) as unknown as Edge[];
           this.graph = buildWalkingGraph(rows);
         }
         getGraph(): WalkingGraph {
           if (!this.graph) throw new Error('WalkingGraphLoader not initialized');
           return this.graph;
         }
       }
       ```
       Use the `DbExecutable` duck-type pattern from Phase 2 (`lag-alert.service.ts`) — don't import Db from @wonderwaltz/db directly.

       - Assert the returned graph resolves known node pairs correctly.
    5. Write `packages/solver/tests/walking-graph-preload.test.ts` (unit): a pure-TS proof that given a graph and an edge list, repeated `shortestPath()` calls do not mutate the graph and are pure.
