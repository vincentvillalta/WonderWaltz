  </behavior>
  <action>
    1. Create `apps/api/src/auth/users.service.ts`:
       - Inject DB_TOKEN
       - `getUserProfile(userId: string)`: SELECT id, email, is_anonymous, created_at FROM users WHERE id = $1 AND deleted_at IS NULL
       - `getTripsCount(userId: string)`: SELECT COUNT(*) FROM trips WHERE user_id = $1 AND deleted_at IS NULL — used by anonymous trip limit guard (Plan 05)

    2. Replace the 501 stub in `users.controller.ts`:
       - Apply `@UseGuards(SupabaseAuthGuard)` to the controller
       - `GET /v1/users/me`: extract `request.user.id`, call `usersService.getUserProfile()`, return UserMeDto. Return 404 if null
       - Keep existing OpenAPI decorators, remove 501 status

    3. Add upgrade endpoint to `auth.controller.ts`:
       - `POST /v1/auth/upgrade` — this endpoint is called AFTER the client completes the OAuth flow via Supabase SDK's `linkIdentity`. The client sends the new JWT. The backend:
         a. Applies `@UseGuards(SupabaseAuthGuard)` — validates the new merged JWT
         b. Reads `request.user` (which now has `isAnonymous: false` and `email` set, since linkIdentity already completed client-side)
         c. Syncs to public.users: `UPDATE users SET is_anonymous = false, email = $1, updated_at = NOW() WHERE id = $2`
         d. Returns 200 with `{ upgraded: true, user_id }`
       - If the user is already non-anonymous, return 200 idempotently (no error)
       - Add DTO: `UpgradeResponseDto { upgraded: boolean; user_id: string }`
       - Per CONTEXT.md locked decision: if merge fails, error message is "This account is already linked to another WonderWaltz account." — but this error surfaces client-side from Supabase SDK, not from our backend. Backend only handles the post-merge sync

    4. Update `auth.module.ts`: add UsersService to providers

    5. Create `apps/api/src/auth/users.service.spec.ts` with tests for all behaviors
  </action>
