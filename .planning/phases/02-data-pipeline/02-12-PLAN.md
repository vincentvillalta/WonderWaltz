    # Phase 2 Data Pipeline — Deployment Runbook

    ## Railway Worker Service Setup

    1. In Railway dashboard: add a new service in the WonderWaltz project
       - Name: `worker`
       - Source: same GitHub repo as `api` service
       - Root directory: `apps/api`
       - Build command: `pnpm run build` (same as api)
       - Start command: `node dist/worker.js`

    2. Copy environment variables from `api` service:
       - DATABASE_URL
       - REDIS_URL
       - SENTRY_DSN_API

    3. Add new environment variables:
       - OPENWEATHER_API_KEY — from OpenWeather dashboard
       - SLACK_ALERT_WEBHOOK_URL — from Slack incoming webhook

    4. Deploy worker service

    ## Verification Steps (run after deploy)

    1. Wait 6 minutes after deploy (one full 5-min polling cycle + buffer)

    2. Check wait_times_history is receiving rows:
       ```sql
       SELECT COUNT(*), MAX(fetched_at) FROM wait_times_history
       WHERE fetched_at > now() - INTERVAL '10 minutes';
       ```
       Expected: COUNT > 0, MAX(fetched_at) within 6 minutes of now()

    3. Check Redis keys exist:
       In Railway Redis CLI or Upstash console:
       ```
       KEYS wait:*
       ```
       Expected: multiple keys matching WDW attraction UUIDs

    4. Check a specific key freshness:
       ```
       GET wait:{any_uuid_from_step_3}
       ```
       Expected: { minutes: N, fetched_at: <within last 5min>, source: "queue-times", is_stale: false }

    5. Check pg_cron is scheduled (run after 1 hour):
       ```sql
       SELECT * FROM cron.job_run_details ORDER BY start_time DESC LIMIT 5;
       ```

    ## Ingestion Clock Start
    Record the timestamp when worker first deployed and verification passes.
    This is t=0 for the 8-week accumulation gate.

    Update docs/ops/PROVISIONING_STATE.md:
    - Mark Railway worker service as provisioned and deployed
    - Record ingestion clock start timestamp
    - Mark Phase 2 as complete
